# Day 008 - Professional Commit Messages and Pull Requests

## Lesson - Commit Message Convention, Pull Requests, and Squash-Merge Divergence

What happened:

- Created a branch (`day-008-pr-practice`) to understand the different PR merge strategies and how pull requests work.
- Practiced writing a commit message with the imperative-mood, ≤50-character summary line convention.
- Opened a real pull request on GitHub, reviewed the diff in the "Files changed" tab, and merged it using squash-and-merge.

Mistake made along the way (or things I found confusing):

- Had uncertainty at the moment of choosing which merge strategy to use.
- After the squash merge, local `main` and `origin/main` showed as diverged. I was confused about this at first — I think it happened because my first commit ("add test line to git.md") ended up on `main` instead of on a branch, and the squash merge created a new commit on `origin/main` with no shared history connecting back to it. Fixed by confirming the content was already on `origin/main`, then running `git reset --hard origin/main`.

Key takeaway:

- Branches exist so `main` stays clean and I have a safe environment to work in without pushing unwanted or broken code. Pull requests matter because they let the team review changes before they land on `main`.
