# Day 007 - Branching in Git

## Lesson - Branching, Switching, Merging, and Remote Branch Cleanup

What happened:

- Created a local-only branch (`day-007-branching`) to see how work gets isolated from `main` — committed a file to it and confirmed the file did not exist back on `main` until I merged it.

- Created a second branch (`day-007-remote-practice`) to practice the full remote lifecycle: pushed it with `git push -u origin <branch>`, watched it appear on GitHub with `git branch -a`, merged it into `main`, pushed `main`, then deleted the branch both locally and on GitHub.

- Both merges were fast-forwards since `main` never moved while I was working on the branch.

Mistake made along the way (or things I found confusing):

- I tried to skip writing the explanations in my own words and asked for the exact wording to copy instead. Got caught on it, which is a good thing since it's exactly what happened with Day 006 too.

Key takeaway:

- Working directly on `main` is risky: if I commit broken/unfinished code there, it affects the whole project for anyone who pulls it, and if `main` auto-deploys, that broken code goes live to real users. Branches exist to isolate in-progress work until it's actually ready to merge in.
